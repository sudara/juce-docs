\doxysubsection{Detailed Description}
\mbox{\hyperlink{classoboe_1_1_audio_stream_data_callback}{Audio\+Stream\+Data\+Callback}} defines a callback interface for moving data to/from an audio stream using {\ttfamily on\+Audio\+Ready} 2) being alerted when a stream has an error using {\ttfamily on\+Error$\ast$} methods

A buffer is ready for processing.

For an output stream, this function should render and write num\+Frames of data in the stream\textquotesingle{}s current data format to the audio\+Data buffer.

For an input stream, this function should read and process num\+Frames of data from the audio\+Data buffer.

The audio data is passed through the buffer. So do NOT call read() or write() on the stream that is making the callback.

Note that num\+Frames can vary unless \mbox{\hyperlink{classoboe_1_1_audio_stream_builder_a206266d7d6c66ff5065fe0cfcc4ba436}{Audio\+Stream\+Builder\+::set\+Frames\+Per\+Callback()}} is called.

Also note that this callback function should be considered a \char`\"{}real-\/time\char`\"{} function. It must not do anything that could cause an unbounded delay because that can cause the audio to glitch or pop.

These are things the function should NOT do\+: 
\begin{DoxyItemize}
\item allocate memory using, for example, malloc() or new 
\item any file operations such as opening, closing, reading or writing 
\item any network operations such as streaming 
\item use any mutexes or other synchronization primitives 
\item sleep 
\item oboe\+Stream-\/\texorpdfstring{$>$}{>}stop(), pause(), flush() or close() 
\item oboe\+Stream-\/\texorpdfstring{$>$}{>}read() 
\item oboe\+Stream-\/\texorpdfstring{$>$}{>}write() 
\end{DoxyItemize}

The following are OK to call from the data callback\+: 
\begin{DoxyItemize}
\item oboe\+Stream-\/\texorpdfstring{$>$}{>}get$\ast$() 
\item \mbox{\hyperlink{namespaceoboe_a0310f9ed27a82c4b9859beb938737096}{oboe\+::convert\+To\+Text()}} 
\item oboe\+Stream-\/\texorpdfstring{$>$}{>}set\+Buffer\+Size\+In\+Frames() 
\end{DoxyItemize}

If you need to move data, eg. MIDI commands, in or out of the callback function then we recommend the use of non-\/blocking techniques such as an atomic FIFO.
