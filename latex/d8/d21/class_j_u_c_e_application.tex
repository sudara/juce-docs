\doxysubsection{Detailed Description}
An instance of this class is used to specify initialisation and shutdown code for the application.

Any application that wants to run an event loop must declare a subclass of \mbox{\hyperlink{class_j_u_c_e_application_base}{JUCEApplication\+Base}} or \mbox{\hyperlink{class_j_u_c_e_application}{JUCEApplication}}, and implement its various pure virtual methods.

It then needs to use the START\+\_\+\+JUCE\+\_\+\+APPLICATION macro somewhere in a CPP file to declare an instance of this class and generate suitable platform-\/specific boilerplate code to launch the app.

Note that this class is derived from \mbox{\hyperlink{class_j_u_c_e_application_base}{JUCEApplication\+Base}}, which contains most of the useful methods and functionality. This derived class is here simply as a convenient way to also inherit from an \mbox{\hyperlink{class_application_command_target}{Application\+Command\+Target}}, and to implement default versions of some of the pure virtual base class methods. But you can derive your app object directly from \mbox{\hyperlink{class_j_u_c_e_application_base}{JUCEApplication\+Base}} if you want to, and by doing so can avoid having a dependency on the juce\+\_\+gui\+\_\+basics module.

This must provide details about one of the commands that this target can perform.

This will be called with one of the command IDs that the target provided in its \mbox{\hyperlink{class_j_u_c_e_application_a72ea0c210591a07bee0bb47a1a3a0db2}{get\+All\+Commands()}} methods.

It should fill-\/in all appropriate fields of the \mbox{\hyperlink{struct_application_command_info}{Application\+Command\+Info}} structure with suitable information about the command. (The command\+ID field will already have been filled-\/in by the caller).

The easiest way to set the info is using the \mbox{\hyperlink{struct_application_command_info_a3e9cd7e72fbf617a0d70d6be3d513b82}{Application\+Command\+Info\+::set\+Info()}} method to set all the fields at once.

If the command is currently inactive for some reason, this method must use \mbox{\hyperlink{struct_application_command_info_a68a043a95e058beeeddfba2441ee7306}{Application\+Command\+Info\+::set\+Active()}} to make that clear, (or it should set the is\+Disabled bit of the \mbox{\hyperlink{struct_application_command_info_a45fa351a09f7a6df3e76fe466c476905}{Application\+Command\+Info\+::flags}} field).

Any default key-\/presses for the command should be appended to the \mbox{\hyperlink{struct_application_command_info_acae1903bdfd8214162adb8326d1ee9da}{Application\+Command\+Info\+::default\+Keypresses}} field.

Note that if you change something that affects the status of the commands that would be returned by this method (e.\+g. something that makes some commands active or inactive), you should call \mbox{\hyperlink{class_application_command_manager_a91fef61705d98da881276733e304e7af}{Application\+Command\+Manager\+::command\+Status\+Changed()}} to cause the manager to refresh its status.

Checks whether multiple instances of the app are allowed.

If your application class returns true for this, more than one instance is permitted to run (except on OSX where the OS automatically stops you launching a second instance of an app without explicitly starting it from the command-\/line).

If it\textquotesingle{}s false, the second instance won\textquotesingle{}t start, but you will still get a callback to \mbox{\hyperlink{class_j_u_c_e_application_adf79c71599bd3848e3d676ed4d95424e}{another\+Instance\+Started()}} to tell you about this -\/ which gives you a chance to react to what the user was trying to do.

If you want to send a quit signal to your app, this is the correct method to call, because it means that requests that come from the system get handled in the same way as those from your own application code.
