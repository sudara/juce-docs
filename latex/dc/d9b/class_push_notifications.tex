\doxysubsection{Detailed Description}
Singleton class responsible for push notifications functionality. Both remote and local notifications are supported. To get information about notifications, register a listener on your application startup. It is best to register the listener as soon as possible, because your application can be launched from a push notification too.

To send a local notification create an instance of \mbox{\hyperlink{struct_push_notifications_1_1_notification}{Notification}}, fill the necessary fields and call \mbox{\hyperlink{class_push_notifications_a7caa92ac2697cdaf137913bd91554dc6}{Push\+Notifications\+::send\+Local\+Notification()}}. When receiving local or remote notifications, inspect the \mbox{\hyperlink{struct_push_notifications_1_1_notification}{Notification}}\textquotesingle{}s fields for notification details. Bear in mind that some fields will not be available when receiving a remote notification.

Initialises push notifications on current device with the settings provided. Call this on your application startup and on i\+OS the first time the application starts, a user will be presented with a permission request dialog to give push notifications permission. Once a user responds, \mbox{\hyperlink{struct_push_notifications_1_1_listener_a7d2d2b7d2f784e18030e9c16893d9c1c}{Listener\+::notification\+Settings\+Received()}} will be called so that you can check what permissions where actually granted. The listener callback will be called on each subsequent startup too (provided you called \mbox{\hyperlink{class_push_notifications_a955877ef0523bcc0d5d2c99dfc3c7271}{request\+Permissions\+With\+Settings()}} on previous application run). This way you can check what are current push notifications permissions.

Note that settings are currently only used on i\+OS. When calling on other platforms, \mbox{\hyperlink{struct_push_notifications_1_1_settings}{Settings}} with no categories and all allow$\ast$ flags set to true will be received in \mbox{\hyperlink{struct_push_notifications_1_1_listener_a7d2d2b7d2f784e18030e9c16893d9c1c}{Listener\+::notification\+Settings\+Received()}}.

You can also call \mbox{\hyperlink{class_push_notifications_a3ff861e614e0e2ddd7f8a7d21bc2234f}{request\+Settings\+Used()}} to explicitly ask for current settings. \mbox{\Hypertarget{class_push_notifications_a3ff861e614e0e2ddd7f8a7d21bc2234f}\label{class_push_notifications_a3ff861e614e0e2ddd7f8a7d21bc2234f}}

Sends an asynchronous request to retrieve current settings that are currently in use. These can be exactly the same as used in \mbox{\hyperlink{class_push_notifications_a955877ef0523bcc0d5d2c99dfc3c7271}{request\+Permissions\+With\+Settings()}}, but depending on user\textquotesingle{}s subsequent changes in OS settings, the actual current settings may be different (e.\+g. user might have later decided to disable sounds).

Note that settings are currently only used on i\+OS and partially on OSX.

On OSX, only allow$\ast$ flags are used and they refer to remote notifications only. For local notifications, refer to System Preferences.

When calling this function on other platforms, \mbox{\hyperlink{struct_push_notifications_1_1_settings}{Settings}} with no categories and all allow$\ast$ flags set to true will be received in \mbox{\hyperlink{struct_push_notifications_1_1_listener_a7d2d2b7d2f784e18030e9c16893d9c1c}{Listener\+::notification\+Settings\+Received()}}. \mbox{\Hypertarget{class_push_notifications_a7caa92ac2697cdaf137913bd91554dc6}\label{class_push_notifications_a7caa92ac2697cdaf137913bd91554dc6}}
