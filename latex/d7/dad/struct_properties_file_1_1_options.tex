This can be called to suggest a file that should be used, based on the values in this structure.

So on a Mac, this will return a file called\+: $\sim$/\+Library/\mbox{[}osx\+Library\+Sub\+Folder\mbox{]}/\mbox{[}folder\+Name\mbox{]}/\mbox{[}application\+Name\mbox{]}.\mbox{[}filename\+Suffix\mbox{]}

On Windows it\textquotesingle{}ll return something like\+: C\+:\textbackslash{}Documents and Settings\textbackslash{}username\textbackslash{}Application Data\textbackslash{}\mbox{[}folder\+Name\mbox{]}\textbackslash{}\mbox{[}application\+Name\mbox{]}.\mbox{[}filename\+Suffix\mbox{]}

On Linux it\textquotesingle{}ll return $\sim$/\mbox{[}folder\+Name\mbox{]}/\mbox{[}application\+Name\mbox{]}.\mbox{[}filename\+Suffix\mbox{]}

If the folder\+Name variable is empty, it\textquotesingle{}ll use the app name for this (or omit the folder name on the Mac).

The paths will also vary depending on whether common\+To\+All\+Users is true.

{\footnotesize\ttfamily \mbox{\hyperlink{class_string}{String}} Properties\+File\+::\+Options\+::osx\+Library\+Sub\+Folder}

If you\textquotesingle{}re using properties files on a Mac, you must set this value -\/ failure to do so will cause a runtime assertion.

The \mbox{\hyperlink{class_properties_file}{Properties\+File}} class always used to put its settings files in \char`\"{}\+Library/\+Preferences\char`\"{}, but Apple have changed their advice, and now stipulate that settings should go in \char`\"{}\+Library/\+Application Support\char`\"{}.

Because older apps would be broken by a silent change in this class\textquotesingle{}s behaviour, you must now explicitly set the osx\+Library\+Sub\+Folder value to indicate which path you want to use.

In newer apps, you should always set this to \char`\"{}\+Application Support\char`\"{} or \char`\"{}\+Application Support/\+Your\+Sub\+Folder\+Name\char`\"{}.

If your app needs to load settings files that were created by older versions of juce and you want to maintain backwards-\/compatibility, then you can set this to \char`\"{}\+Preferences\char`\"{}. But.. for better Apple-\/compliance, the recommended approach would be to write some code that finds your old settings files in $\sim$/\+Library/\+Preferences, moves them to $\sim$/\+Library/\+Application Support, and then uses the new path. \mbox{\Hypertarget{struct_properties_file_1_1_options_a65881e8a7c6e90fbaca03dd2f00a34a7}\label{struct_properties_file_1_1_options_a65881e8a7c6e90fbaca03dd2f00a34a7}}
