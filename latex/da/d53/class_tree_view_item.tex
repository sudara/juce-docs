\doxysubsection{Detailed Description}
An item in a \mbox{\hyperlink{class_tree_view}{Tree\+View}}.

A \mbox{\hyperlink{class_tree_view_item}{Tree\+View\+Item}} can either be a leaf-\/node in the tree, or it can contain its own sub-\/items.

To implement an item that contains sub-\/items, override the \mbox{\hyperlink{class_tree_view_item_a632fcf45aa68c0880765b21a267c0d74}{item\+Openness\+Changed()}} method so that when it is opened, it adds the new sub-\/items to itself using the add\+Sub\+Item method. Depending on the nature of the item it might choose to only do this the first time it\textquotesingle{}s opened, or it might want to refresh itself each time. It also has the option of deleting its sub-\/items when it is closed, or leaving them in place.

Creates a component that will be used to represent this item.

You don\textquotesingle{}t have to implement this method -\/ if it returns nullptr then no component will be used for the item, and you can just draw it using the \mbox{\hyperlink{class_tree_view_item_a31587d447a9622b647ac9d8d6b9e63be}{paint\+Item()}} callback. But if you do return a component, it will be positioned in the \mbox{\hyperlink{class_tree_view}{Tree\+View}} so that it can be used to represent this item.

The component returned will be managed by the \mbox{\hyperlink{class_tree_view}{Tree\+View}} and will be deleted later when it goes off the screen or is no longer needed. Its position and size will be completely managed by the tree, so don\textquotesingle{}t attempt to move it around.

Something you may want to do with your component is to give it a pointer to the \mbox{\hyperlink{class_tree_view}{Tree\+View}} that created it. This is perfectly safe, and there\textquotesingle{}s no danger of it becoming a dangling pointer because the \mbox{\hyperlink{class_tree_view}{Tree\+View}} will always delete the component before it is itself deleted.

As long as you stick to these rules you can return whatever kind of component you like. It\textquotesingle{}s most useful if you\textquotesingle{}re doing things like drag-\/and-\/drop of items, or want to use a \mbox{\hyperlink{class_label}{Label}} component to edit item names, etc.

To identify node objects it uses the \mbox{\hyperlink{class_tree_view_item_aea8e4339244ee958fd91080791ed239d}{Tree\+View\+Item\+::get\+Unique\+Name()}} method to create named paths. This means that the same state of open/closed nodes can be restored to a completely different instance of the tree, as long as it contains nodes whose unique names are the same.

You\textquotesingle{}d normally want to use \mbox{\hyperlink{class_tree_view_a7d7c90a3d4e7e1e76bef68b7d0787302}{Tree\+View\+::get\+Openness\+State()}} rather than call it for a specific item, but this can be handy if you need to briefly save the state for a section of the tree.

Note that if all nodes of the tree are in their default state, then this may return a nullptr.

By default this is false, which means that when the \mbox{\hyperlink{class_tree_view_item_a31587d447a9622b647ac9d8d6b9e63be}{paint\+Item()}} method is called, its graphics context is clipped to only allow drawing within the item\textquotesingle{}s rectangle. It\textquotesingle{}s mostly useful if you want to draw a wider bar behind the highlighted item. \mbox{\Hypertarget{class_tree_view_item_a5d74053652a52f66f97aa611a2854c8e}\label{class_tree_view_item_a5d74053652a52f66f97aa611a2854c8e}}
