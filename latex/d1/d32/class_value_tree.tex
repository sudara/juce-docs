\doxysubsection{Detailed Description}
A powerful tree structure that can be used to hold free-\/form data, and which can handle its own undo and redo behaviour.

A \mbox{\hyperlink{class_value_tree}{Value\+Tree}} contains a list of named properties as var objects, and also holds any number of sub-\/trees.

Create \mbox{\hyperlink{class_value_tree}{Value\+Tree}} objects on the stack, and don\textquotesingle{}t be afraid to copy them around, as they\textquotesingle{}re simply a lightweight reference to a shared data container. Creating a copy of another \mbox{\hyperlink{class_value_tree}{Value\+Tree}} simply creates a new reference to the same underlying object -\/ to make a separate, deep copy of a tree you should explicitly call \mbox{\hyperlink{class_value_tree_aa0bb4e88b31f73f802799d6ebb324878}{create\+Copy()}}.

Each \mbox{\hyperlink{class_value_tree}{Value\+Tree}} has a type name, in much the same way as an \mbox{\hyperlink{class_xml_element}{Xml\+Element}} has a tag name, and much of the structure of a \mbox{\hyperlink{class_value_tree}{Value\+Tree}} is similar to an \mbox{\hyperlink{class_xml_element}{Xml\+Element}} tree. You can convert a \mbox{\hyperlink{class_value_tree}{Value\+Tree}} to and from an \mbox{\hyperlink{class_xml_element}{Xml\+Element}}, and as long as the XML doesn\textquotesingle{}t contain text elements, the conversion works well and makes a good serialisation format. They can also be serialised to a binary format, which is very fast and compact.

All the methods that change data take an optional \mbox{\hyperlink{class_undo_manager}{Undo\+Manager}}, which will be used to track any changes to the object. For this to work, you have to be careful to consistently always use the same \mbox{\hyperlink{class_undo_manager}{Undo\+Manager}} for all operations to any sub-\/tree inside the tree.

A \mbox{\hyperlink{class_value_tree}{Value\+Tree}} can only be a child of one parent at a time, so if you\textquotesingle{}re moving one from one tree to another, be careful to always remove it first, before adding it. This could also mess up your undo/redo chain, so be wary! In a debug build you should hit assertions if you try to do anything dangerous, but there are still plenty of ways it could go wrong.

Note that although the children in a tree have a fixed order, the properties are not guaranteed to be stored in any particular order, so don\textquotesingle{}t expect that a property\textquotesingle{}s index will correspond to the order in which the property was added, or that it will remain constant when other properties are added or removed.

Listeners can be added to a \mbox{\hyperlink{class_value_tree}{Value\+Tree}} to be told when properties change and when sub-\/trees are added or removed.

In general, you\textquotesingle{}d never want to add a listener to a local stack-\/based \mbox{\hyperlink{class_value_tree}{Value\+Tree}}, and would usually add one to a member variable.
