\doxysubsection{Detailed Description}
A special type of \mbox{\hyperlink{class_audio_processor}{Audio\+Processor}} that can live inside an \mbox{\hyperlink{class_audio_processor_graph}{Audio\+Processor\+Graph}} in order to use the audio that comes into and out of the graph itself.

If you create an \mbox{\hyperlink{class_audio_processor_graph_1_1_audio_graph_i_o_processor}{Audio\+Graph\+IOProcessor}} in \char`\"{}input\char`\"{} mode, it will act as a node in the graph which delivers the audio that is coming into the parent graph. This allows you to stream the data to other nodes and process the incoming audio.

Likewise, one of these in \char`\"{}output\char`\"{} mode can be sent data which it will add to the sum of data being sent to the graph\textquotesingle{}s output.

{\footnotesize\ttfamily \mbox{\hyperlink{class_audio_processor_editor}{Audio\+Processor\+Editor}} $\ast$ juce\+::\+Audio\+Processor\+Graph\+::\+Audio\+Graph\+IOProcessor\+::create\+Editor (\begin{DoxyParamCaption}{ }\end{DoxyParamCaption})\hspace{0.3cm}{\ttfamily [override]}, {\ttfamily [virtual]}}

Creates the processor\textquotesingle{}s GUI.

This can return nullptr if you want a GUI-\/less processor, in which case the host may create a generic UI that lets the user twiddle the parameters directly.

If you do want to pass back a component, the component should be created and set to the correct size before returning it. If you implement this method, you must also implement the \mbox{\hyperlink{class_audio_processor_graph_1_1_audio_graph_i_o_processor_a4d13a600a74a5ca708a7fe643f5ed671}{has\+Editor()}} method and make it return true.

Remember not to do anything silly like allowing your processor to keep a pointer to the component that gets created -\/ it could be deleted later without any warning, which would make your pointer into a dangler. Use the \mbox{\hyperlink{class_audio_processor_aaa35e308aa5fde974e1a922bf4e7c42d}{get\+Active\+Editor()}} method instead.

The correct way to handle the connection between an editor component and its processor is to use something like a \mbox{\hyperlink{class_change_broadcaster}{Change\+Broadcaster}} so that the editor can register itself as a listener, and be told when a change occurs. This lets them safely unregister themselves when they are deleted.

Here are a few things to bear in mind when writing an editor\+:


\begin{DoxyItemize}
\item Initially there won\textquotesingle{}t be an editor, until the user opens one, or they might not open one at all. Your processor mustn\textquotesingle{}t rely on it being there.
\item An editor object may be deleted and a replacement one created again at any time.
\item It\textquotesingle{}s safe to assume that an editor will be deleted before its processor.
\end{DoxyItemize}

{\footnotesize\ttfamily \mbox{\hyperlink{png_8h_aa8c59027f9ab2769342f248709d68d17}{void}} Audio\+Processor\+Graph\+::\+Audio\+Graph\+IOProcessor\+::process\+Block (\begin{DoxyParamCaption}\item[{\mbox{\hyperlink{class_audio_buffer}{Audio\+Buffer}}$<$ double $>$ \&}]{buffer,  }\item[{\mbox{\hyperlink{class_midi_buffer}{Midi\+Buffer}} \&}]{midi\+Messages }\end{DoxyParamCaption})\hspace{0.3cm}{\ttfamily [override]}, {\ttfamily [virtual]}}

Renders the next block.

When this method is called, the buffer contains a number of channels which is at least as great as the maximum number of input and output channels that this processor is using. It will be filled with the processor\textquotesingle{}s input data and should be replaced with the processor\textquotesingle{}s output.

So for example if your processor has a combined total of 2 input channels and 4 output channels, then the buffer will contain 4 channels, the first two being filled with the input data. Your processor should read these, do its processing, and replace the contents of all 4 channels with its output.

Or if your processor has 5 inputs and 2 outputs, the buffer will have 5 channels, all filled with data, and your processor should overwrite the first 2 of these with its output. But be VERY careful not to write anything to the last 3 channels, as these might be mapped to memory that the host assumes is read-\/only!

If your plug-\/in has more than one input or output buses then the buffer passed to the process\+Block methods will contain a bundle of all channels of each bus. Use get\+Bus\+Buffer to obtain a audio buffer for a particular bus.

Note that if you have more outputs than inputs, then only those channels that correspond to an input channel are guaranteed to contain sensible data -\/ e.\+g. in the case of 2 inputs and 4 outputs, the first two channels contain the input, but the last two channels may contain garbage, so you should be careful not to let this pass through without being overwritten or cleared.

Also note that the buffer may have more channels than are strictly necessary, but you should only read/write from the ones that your processor is supposed to be using.

If your plugin uses buses, then you should use \mbox{\hyperlink{class_audio_processor_a733377c6d541c099dc5857be77cd1af1}{get\+Bus\+Buffer()}} or \mbox{\hyperlink{class_audio_processor_af9de177654fe95cc9de951df15537c24}{get\+Channel\+Index\+In\+Process\+Block\+Buffer()}} to find out which of the input and output channels correspond to which of the buses.

The number of samples in these buffers is NOT guaranteed to be the same for every callback, and may be more or less than the estimated value given to \mbox{\hyperlink{class_audio_processor_graph_1_1_audio_graph_i_o_processor_a10cbbf7505be97b0e3d8235004d6c925}{prepare\+To\+Play()}}. Your code must be able to cope with variable-\/sized blocks, or you\textquotesingle{}re going to get clicks and crashes!

Also note that some hosts will occasionally decide to pass a buffer containing zero samples, so make sure that your algorithm can deal with that!

If the processor is receiving a MIDI input, then the midi\+Messages array will be filled with the MIDI messages for this block. Each message\textquotesingle{}s timestamp will indicate the message\textquotesingle{}s time, as a number of samples from the start of the block.

Any messages left in the MIDI buffer when this method has finished are assumed to be the processor\textquotesingle{}s MIDI output. This means that your processor should be careful to clear any incoming messages from the array if it doesn\textquotesingle{}t want them to be passed-\/on.

If you have implemented the get\+Bypass\+Parameter method, then you need to check the value of this parameter in this callback and bypass your processing if the parameter has a non-\/zero value.

Note that when calling this method as a host, the result may still be bypassed as the parameter that controls the bypass may be non-\/zero.

Be very careful about what you do in this callback -\/ it\textquotesingle{}s going to be called by the audio thread, so any kind of interaction with the UI is absolutely out of the question. If you change a parameter in here and need to tell your UI to update itself, the best way is probably to inherit from a \mbox{\hyperlink{class_change_broadcaster}{Change\+Broadcaster}}, let the UI components register as listeners, and then call \mbox{\hyperlink{class_change_broadcaster_a10f01d1c0a7fde85355321985a682cbd}{send\+Change\+Message()}} inside the \mbox{\hyperlink{class_audio_processor_graph_1_1_audio_graph_i_o_processor_ae79b4debef5e39306aea4dfd6dff7580}{process\+Block()}} method to send out an asynchronous message. You could also use the \mbox{\hyperlink{class_async_updater}{Async\+Updater}} class in a similar way.
