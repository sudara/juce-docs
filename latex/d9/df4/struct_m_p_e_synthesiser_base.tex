\doxysubsection{Detailed Description}
Derive from this class to create a basic audio generator capable of MPE. Implement the callbacks of \mbox{\hyperlink{class_m_p_e_instrument_1_1_listener}{MPEInstrument\+::\+Listener}} (note\+Added, note\+Pressure\+Changed etc.) to let your audio generator know that MPE notes were triggered, modulated, or released. What to do inside them, and how that influences your audio generator, is up to you!

This class uses an instance of \mbox{\hyperlink{class_m_p_e_instrument}{MPEInstrument}} internally to handle the MPE note state logic.

This class is a very low-\/level base class for an MPE instrument. If you need something more sophisticated, have a look at \mbox{\hyperlink{class_m_p_e_synthesiser}{MPESynthesiser}}. This class extends \mbox{\hyperlink{struct_m_p_e_synthesiser_base}{MPESynthesiser\+Base}} by adding the concept of voices that can play notes, a voice stealing algorithm, and much more.

Handle incoming MIDI events (called from render\+Next\+Block).

The default implementation provided here simply forwards everything to \mbox{\hyperlink{class_m_p_e_instrument_a2d4ea1f5fcd64d8feeac46ee9e5db712}{MPEInstrument\+::process\+Next\+Midi\+Event}}, where it is used to update the MPE notes, zones etc. MIDI messages not relevant for MPE are ignored.

This method can be overridden if you need to do custom MIDI handling on top of MPE. The \mbox{\hyperlink{class_m_p_e_synthesiser}{MPESynthesiser}} class overrides this to implement callbacks for MIDI program changes and non-\/\+MPE-\/related MIDI controller messages.

Creates the next block of audio output.

Call this to make sound. This will chop up the \mbox{\hyperlink{class_audio_buffer}{Audio\+Buffer}} into sub\+Block pieces separated by events in the MIDI buffer, and then call render\+Next\+Sub\+Block on each one of them. In between you will get calls to note\+Added/\+Changed/\+Finished, where you can update parameters that depend on those notes to use for your audio rendering.

{\footnotesize\ttfamily \mbox{\hyperlink{png_8h_aa8c59027f9ab2769342f248709d68d17}{void}} MPESynthesiser\+Base\+::set\+Minimum\+Rendering\+Subdivision\+Size (\begin{DoxyParamCaption}\item[{\mbox{\hyperlink{group__juce__audio__plugin__client-_unity_ga966b05a42d98c7296097e073c627dcd0}{int}}}]{num\+Samples,  }\item[{bool}]{should\+Be\+Strict = {\ttfamily \mbox{\hyperlink{ordinals_8h_a65e9886d74aaee76545e83dd09011727}{false}}} }\end{DoxyParamCaption})\hspace{0.3cm}{\ttfamily [noexcept]}}

Sets a minimum limit on the size to which audio sub-\/blocks will be divided when rendering.

When rendering, the audio blocks that are passed into \mbox{\hyperlink{struct_m_p_e_synthesiser_base_a7dcd22df95ad83ee04d8ebc5ae605176}{render\+Next\+Block()}} will be split up into smaller blocks that lie between all the incoming midi messages, and it is these smaller sub-\/blocks that are rendered with multiple calls to render\+Voices().

Obviously in a pathological case where there are midi messages on every sample, then render\+Voices() could be called once per sample and lead to poor performance, so this setting allows you to set a lower limit on the block size.

The default setting is 32, which means that midi messages are accurate to about \texorpdfstring{$<$}{<} 1ms accuracy, which is probably fine for most purposes, but you may want to increase or decrease this value for your synth.

If should\+Be\+Strict is true, the audio sub-\/blocks will strictly never be smaller than num\+Samples.

If should\+Be\+Strict is false (default), the first audio sub-\/block in the buffer is allowed to be smaller, to make sure that the first MIDI event in a buffer will always be sample-\/accurate (this can sometimes help to avoid quantisation or phasing issues). \mbox{\Hypertarget{struct_m_p_e_synthesiser_base_a05a40ad0af91f7802f48c1c95182b62c}\label{struct_m_p_e_synthesiser_base_a05a40ad0af91f7802f48c1c95182b62c}}
