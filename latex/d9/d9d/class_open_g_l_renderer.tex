\doxysubsubsection{\texorpdfstring{renderOpenGL()}{renderOpenGL()}}
{\footnotesize\ttfamily virtual \mbox{\hyperlink{png_8h_aa8c59027f9ab2769342f248709d68d17}{void}} Open\+GLRenderer\+::render\+Open\+GL (\begin{DoxyParamCaption}{ }\end{DoxyParamCaption})\hspace{0.3cm}{\ttfamily [pure virtual]}}

Called when you should render the next open\+GL frame.

Note that this callback will be made on a background thread.

If the context is attached to a component in order to do component rendering, then the \mbox{\hyperlink{class_message_manager}{Message\+Manager}} will be locked when this callback is made.

If no component rendering is being done, then the \mbox{\hyperlink{class_message_manager}{Message\+Manager}} will not be locked, and you\textquotesingle{}ll need to make sure your code is thread-\/safe in any interactions it has with your GUI classes.

For information about how to trigger a render callback, see \mbox{\hyperlink{class_open_g_l_context_acc0fc11c02ebbe4af1cc234eb5d8b84a}{Open\+GLContext\+::trigger\+Repaint()}} and \mbox{\hyperlink{class_open_g_l_context_a7c305575ef3337ae64940db3c00abdc6}{Open\+GLContext\+::set\+Continuous\+Repainting()}}.

IMPORTANT\+: Never take a \mbox{\hyperlink{class_message_manager_lock}{Message\+Manager\+Lock}} inside this function! On mac\+OS, the Open\+GL context will be locked for the duration of this call. The main thread may also attempt to interact with the Open\+GL context at any time, which will also require locking the Open\+GL context. As a result, taking a \mbox{\hyperlink{class_message_manager_lock}{Message\+Manager\+Lock}} inside \mbox{\hyperlink{class_open_g_l_renderer_a34077b4eaeb3207814b9e4302f724912}{render\+Open\+GL()}} may cause a hierarchical deadlock.
