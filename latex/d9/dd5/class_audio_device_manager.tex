\doxysubsection{Detailed Description}
Manages the state of some audio and midi i/o devices.

This class keeps tracks of a currently-\/selected audio device, through with which it continuously streams data from an audio callback, as well as one or more midi inputs.

The idea is that your application will create one global instance of this object, and let it take care of creating and deleting specific types of audio devices internally. So when the device is changed, your callbacks will just keep running without having to worry about this.

The manager can save and reload all of its device settings as XML, which makes it very easy for you to save and reload the audio setup of your application.

And to make it easy to let the user change its settings, there\textquotesingle{}s a component to do just that -\/ the \mbox{\hyperlink{class_audio_device_selector_component}{Audio\+Device\+Selector\+Component}} class, which contains a set of device selection/sample-\/rate/latency controls.

To use an \mbox{\hyperlink{class_audio_device_manager}{Audio\+Device\+Manager}}, create one, and use \mbox{\hyperlink{class_audio_device_manager_a4c6d8a76a270a6e4ca0537093f447b44}{initialise()}} to set it up. Then call \mbox{\hyperlink{class_audio_device_manager_acf3977dcc83f22b7f51091d7ff7b8aff}{add\+Audio\+Callback()}} to register your audio callback with it, and use that to process your audio data.

The manager also acts as a handy hub for incoming midi messages, allowing a listener to register for messages from either a specific midi device, or from whatever the current default midi input device is. The listener then doesn\textquotesingle{}t have to worry about re-\/registering with different midi devices if they are changed or deleted.

And yet another neat trick is that amount of CPU time being used is measured and available with the \mbox{\hyperlink{class_audio_device_manager_a79563c9d2cc87d059d8b101515e8dee1}{get\+Cpu\+Usage()}} method.

The \mbox{\hyperlink{class_audio_device_manager}{Audio\+Device\+Manager}} is a \mbox{\hyperlink{class_change_broadcaster}{Change\+Broadcaster}}, and will send a change message to listeners whenever one of its settings is changed.

Registers a listener for callbacks when midi events arrive from a midi input.

Enables or disables a midi input device.

The list of devices can be obtained with the \mbox{\hyperlink{class_midi_input_a5696a5c974be7e2551701dabfcaabe63}{Midi\+Input\+::get\+Available\+Devices()}} method.

Any incoming messages from enabled input devices will be forwarded on to all the listeners that have been registered with the \mbox{\hyperlink{class_audio_device_manager_a3d3cabde443512bedcb868091a23a8ee}{add\+Midi\+Input\+Device\+Callback()}} method. They can either register for messages from a particular device, or from just the \char`\"{}default\char`\"{} midi input.

Routing the midi input via an \mbox{\hyperlink{class_audio_device_manager}{Audio\+Device\+Manager}} means that when a listener registers for the default midi input, this default device can be changed by the manager without the listeners having to know about it or re-\/register.

It also means that a listener can stay registered for a midi input that is disabled or not present, so that when the input is re-\/enabled, the listener will start receiving messages again.
